# 学生端产品方案

## 1. 方案定位

学生端是教师端 MVP 的配套使用端，目标不是建设完整学习平台，而是让教师端的“发布作业 -> 学生提交 -> AI 初评 -> 教师复核 -> 发布结果 -> 学生查看”闭环真正跑通。

第一版建议做成轻量 Web 端，优先适配桌面浏览器和手机浏览器。学生使用场景更偏移动端和临时访问，因此页面应少层级、强状态提示、减少复杂配置。

## 2. 产品原则

- 学生只看和自己有关的作业、提交和结果。
- 学生端不暴露 AI 初评原始内容。
- 作业提交路径必须短，最多三步完成。
- 每个作业都要明确展示当前状态和下一步动作。
- 提交成功、迟交、退回、结果发布都要有清晰反馈。
- 学生端状态必须能反向驱动教师端统计。

## 3. 范围落地策略

学生端第一版按“先闭环、再增强、后扩展”的方式落地。

### 3.1 先做

先做能让教师端主链路成立的功能：

- 学生登录或身份校验。
- 学生作业列表。
- 作业详情。
- 作业提交。
- 提交成功与提交详情。
- 教师退回后的重新提交。
- 教师发布后的结果查看。
- 学生本人数据权限控制。

### 3.2 不做

第一版不做会把产品扩成完整学习平台的能力：

- 不做课程资料库。
- 不做课堂签到。
- 不做在线考试。
- 不做讨论区。
- 不做学生互评。
- 不做复杂消息中心。
- 不做学习分析。
- 不做成绩申诉。

### 3.3 分层

核心功能必须服务完整闭环，辅助功能只服务体验和效率，后续功能进入迭代池。

| 层级 | 功能 | 处理方式 |
| --- | --- | --- |
| 核心功能 | 登录、作业列表、详情、提交、提交详情、结果查看 | MVP 必须完成 |
| 辅助功能 | 状态筛选、迟交提示、上传进度、提交确认、空状态 | MVP 尽量完成 |
| 后续功能 | 通知中心、申诉、互评、学习分析、资料中心 | MVP 后置 |

### 3.4 MVP 判定

学生端 MVP 是否成立，只看一条链路：

```text
教师发布作业 -> 学生查看作业 -> 学生提交作业 -> 教师看到提交 -> 教师发布结果 -> 学生查看结果
```

如果这条链路能用静态数据和真实接口分别跑通，学生端 MVP 才算完成。

## 4. 信息架构

```text
学生端
├─ 登录 / 访问
├─ 作业列表
│  ├─ 待提交
│  ├─ 已提交
│  ├─ 已退回
│  └─ 已发布结果
├─ 作业详情
├─ 提交作业
├─ 提交详情
└─ 批阅结果
```

## 5. 核心闭环

### 5.1 学生提交闭环

```text
教师发布作业
-> 学生作业列表出现待提交任务
-> 学生查看作业详情
-> 学生填写文本 / 链接 / 上传附件
-> 学生提交
-> 系统生成提交记录
-> 教师端提交数增加
-> AI 初评或人工批阅进入后续流程
```

### 5.2 退回修改闭环

```text
教师退回提交
-> 学生作业列表显示已退回
-> 学生查看退回原因
-> 学生修改内容并重新提交
-> 提交版本增加
-> 教师端重新进入待处理
```

### 5.3 结果查看闭环

```text
教师发布结果
-> 学生作业列表显示已发布结果
-> 学生进入结果页
-> 查看最终分数和教师评语
```

## 6. 业务流程方案

### 6.1 流程总览

学生端业务流程围绕三件事展开：进入作业、完成提交、查看结果。流程设计要保证学生每一步都知道当前状态和下一步动作，同时保证教师端能接收到准确状态。

详细流程图见 `student-side-business-flowcharts.md`，其中包含总业务流程、学生提交流程、教师复核流程、异常分支、状态流转和双端联动流程。

```text
进入产品
-> 身份校验
-> 查看作业列表
-> 查看作业详情
-> 提交作业
-> 查看提交状态
-> 等待教师复核
-> 查看发布结果
```

### 6.2 从进入产品到完成目标

| 步骤 | 学生动作 | 系统响应 | 下一步 |
| --- | --- | --- | --- |
| 进入 | 打开学生端或作业链接 | 判断是否已登录 | 未登录则进入登录页 |
| 登录 | 输入学号、手机号或验证码 | 校验身份，绑定当前学生 | 进入作业列表或目标作业 |
| 找作业 | 查看作业列表和状态 | 展示待提交、已退回、已发布结果 | 进入作业详情 |
| 看要求 | 阅读说明、截止时间、提交方式 | 展示可提交内容和限制 | 进入提交页 |
| 提交 | 填写文本、链接或上传附件 | 校验内容、上传文件、创建提交 | 展示提交成功 |
| 等待 | 查看提交状态 | 展示待批阅、迟交或已退回 | 等待教师处理或重新提交 |
| 看结果 | 点击已发布结果作业 | 展示最终分数和教师评语 | 完成闭环 |

### 6.3 角色协作

| 角色 | 负责内容 | 与学生端关系 |
| --- | --- | --- |
| 学生 | 查看作业、提交作业、查看结果 | 学生端核心操作者 |
| 教师 | 发布作业、退回、复核、发布成绩 | 决定学生端可见作业和结果 |
| 系统 | 权限、状态、文件、同步 | 保证流程稳定运行 |
| AI | 初评建议 | 只服务教师端，不在学生端展示 |

### 6.4 异常处理方案

| 异常 | 触发场景 | 产品处理 |
| --- | --- | --- |
| 未登录 | 直接访问作业链接 | 跳转登录，登录后回到原作业 |
| 无权限 | 学生不属于该课程班 | 显示无权限提示，不展示作业内容 |
| 作业已截止 | 超过截止时间 | 若允许迟交，标记迟交；否则禁止提交 |
| 作业不可见 | 草稿、撤回、删除 | 提示作业暂不可访问 |
| 提交内容为空 | 文本、链接、附件都未填 | 阻止提交并提示至少填写一种 |
| 上传失败 | 附件上传中断 | 保留表单内容，允许重试 |
| 重复提交 | 多次点击提交按钮 | 按钮进入处理中，避免重复创建 |
| 教师未发布结果 | 学生进入结果页 | 显示待批阅，不返回分数评语 |
| 访问他人结果 | 修改链接或 ID | 拦截访问，返回无权限 |

### 6.5 支撑能力清单

#### 必需支撑能力

- 学生身份认证。
- 学生与课程班关系校验。
- 作业可见性控制。
- 提交内容校验。
- 文件上传与附件访问鉴权。
- 提交状态流转。
- 教师端与学生端状态同步。
- 结果发布可见性控制。

#### 体验支撑能力

- 迟交提示。
- 退回原因展示。
- 提交成功反馈。
- 上传进度。
- 空状态。
- 错误重试。
- 登录后回跳。

#### 后续支撑能力

- 消息通知。
- 更正记录。
- 提交版本对比。
- 学生申诉或留言。

## 7. 页面方案

### 7.1 登录 / 访问页

#### 页面目标

让学生快速进入本人作业空间。

#### 推荐方案

MVP 支持两种入口：

- 常规入口：学生通过学号或手机号登录。
- 作业链接入口：学生点击教师分享的作业链接，完成身份校验后进入对应作业。

#### 页面内容

- 系统名称。
- 学号或手机号输入框。
- 验证码或密码输入框。
- 登录按钮。
- 当前访问作业提示。

#### 交互规则

- 从作业链接进入时，登录成功后跳转作业详情。
- 普通登录成功后跳转作业列表。
- 登录过期时保留原目标页面，重新登录后回跳。

### 7.2 作业列表页

#### 页面目标

让学生一眼看到当前最应该处理的作业。

#### 推荐布局

顶部展示学生信息和统计：

- 待提交。
- 已提交。
- 已退回。
- 已发布结果。

下方展示作业列表，按紧急程度排序：

1. 已退回。
2. 临近截止未提交。
3. 待提交。
4. 已提交待批阅。
5. 已发布结果。
6. 历史作业。

#### 作业卡片字段

- 作业标题。
- 课程班。
- 截止时间。
- 状态标签。
- 操作按钮。

#### 状态与按钮

| 状态 | 主按钮 |
| --- | --- |
| 待提交 | 去提交 |
| 临近截止 | 去提交 |
| 已提交 | 查看提交 |
| 迟交 | 查看提交 |
| 已退回 | 修改并重新提交 |
| 已发布结果 | 查看结果 |
| 已截止不可提交 | 查看详情 |

### 7.3 作业详情页

#### 页面目标

让学生明确作业要求和提交条件。

#### 页面模块

- 作业基本信息：标题、课程班、教师、截止时间。
- 作业说明：富文本或普通文本。
- 作业附件：教师上传的说明、模板、评分说明中可对学生公开的部分。
- 提交要求：附件、文本、链接。
- 当前提交状态。
- 底部主操作按钮。

#### 关键设计

- 截止时间要突出。
- 若已逾期但允许迟交，按钮显示“迟交提交”。
- 若教师退回，退回原因放在主内容前方。
- 不展示 AI 评分标准中的内部教师提示。

### 7.4 提交作业页

#### 页面目标

让学生低成本完成提交，并减少错交。

#### 表单模块

- 文本提交。
- 链接提交。
- 附件上传。
- 提交确认区。

#### 提交校验

- 文本、链接、附件至少一种。
- 链接需符合 URL 格式。
- 附件大小和类型需符合限制。
- 上传未完成时不能提交。
- 已截止且不允许迟交时不能提交。

#### 提交后

成功后跳转提交成功页，展示：

- 提交状态。
- 提交时间。
- 提交版本。
- 是否迟交。
- 返回列表和查看提交两个按钮。

### 7.5 提交详情页

#### 页面目标

让学生确认自己提交了什么。

#### 页面模块

- 提交状态。
- 提交时间。
- 提交版本。
- 文本内容。
- 链接。
- 附件。
- 退回原因。

#### 操作

- 查看或下载附件。
- 退回状态下重新提交。

### 7.6 批阅结果页

#### 页面目标

展示教师发布后的最终结果。

#### 页面模块

- 最终分数。
- 教师评语。
- 发布时间。
- 对应提交版本。
- 作业基本信息。

#### 不展示内容

- AI 建议分数。
- AI 原始评语。
- AI 评分依据。
- 教师未发布草稿。

## 8. 状态流转方案

### 8.1 学生提交状态

```mermaid
stateDiagram-v2
  [*] --> "未提交"
  "未提交" --> "已提交": 截止前提交
  "未提交" --> "迟交": 截止后提交
  "已提交" --> "已退回": 教师退回
  "迟交" --> "已退回": 教师退回
  "已退回" --> "已重新提交": 学生重新提交
  "已重新提交" --> "已退回": 教师再次退回
```

### 8.2 学生可见结果状态

```mermaid
stateDiagram-v2
  [*] --> "待批阅"
  "待批阅" --> "已退回": 教师退回
  "待批阅" --> "已发布结果": 教师发布
  "已退回" --> "待批阅": 学生重新提交
  "已发布结果" --> "已更正": 教师更正并重新发布
```

## 9. 与教师端对应关系

| 教师端操作 | 学生端变化 |
| --- | --- |
| 发布作业 | 作业出现在学生列表 |
| 修改作业说明 | 作业详情同步更新 |
| 修改截止时间 | 列表和详情截止时间更新 |
| 关闭提交 | 学生端禁止提交 |
| 退回修改 | 学生端显示退回原因和重新提交入口 |
| 保存批阅草稿 | 学生端仍显示待批阅 |
| 发布结果 | 学生端显示最终分数和评语 |
| 更正成绩 | 学生端显示最新结果，可标记已更正 |

## 10. 数据与接口方案

### 10.1 前端数据模型

学生端页面需要以下聚合数据：

- 当前学生信息。
- 作业列表聚合数据。
- 作业详情。
- 本人提交详情。
- 本人结果详情。

### 10.2 接口映射

| 页面 | 接口 |
| --- | --- |
| 登录页 | `POST /api/auth/student/dev-login` |
| 作业列表 | `GET /api/student/assignments` |
| 作业详情 | `GET /api/student/assignments/{assignment_id}` |
| 提交作业 | `POST /api/student/assignments/{assignment_id}/submit` |
| 提交详情 | `GET /api/student/submissions/{submission_id}` |
| 批阅结果 | `GET /api/student/assignments/{assignment_id}/result` |
| 附件上传 | `POST /api/files/upload-token` |

### 10.3 接口返回原则

- 作业列表接口直接返回学生视角状态，减少前端拼装。
- 结果接口未发布时返回未发布状态，不返回分数。
- 学生端不返回 AI 初评字段。
- 提交接口返回最新 submission，便于立即刷新状态。

## 11. 静态原型建议

为了和现有教师端静态原型打通，建议学生端先做同一个 HTML 文件或同一套静态数据中的学生页面。

### 11.1 建议新增页面 ID

- `student-login`
- `student-dashboard`
- `student-assignment-detail`
- `student-submit`
- `student-submit-success`
- `student-submission-detail`
- `student-result`

### 11.2 建议统一 mockData

在当前教师端原型已有数据基础上扩展：

```js
mockData.students = [
  {
    id: "student-001",
    name: "张同学",
    studentNo: "20260001",
    classIds: ["course-chinese-a"]
  }
];

mockData.studentSubmissions = [
  {
    assignmentId: "paper",
    studentId: "student-001",
    status: "not_submitted",
    submitText: "",
    submitLink: "",
    attachments: [],
    version: 0,
    resultVisible: false
  }
];
```

### 11.3 静态交互优先级

第一优先级：

- 学生端作业列表从 `mockData.assignments` 渲染。
- 点击提交后修改 `studentSubmissions` 状态。
- 教师端提交数和未提交数同步变化。
- 教师发布结果后，学生端结果可见。

第二优先级：

- 退回修改后学生端显示退回原因。
- 重新提交后版本递增。
- 截止后提交标记迟交。

第三优先级：

- 多学生切换。
- 多课程班作业列表。
- 附件上传模拟进度。

## 13. 里程碑建议

### M1 静态闭环

目标：先用统一测试数据跑通页面演示。

范围：

- 学生端页面加入当前 HTML 原型。
- 学生提交后教师端数字变化。
- 教师发布后学生端可查看结果。

验收：

- 不接后端也能演示完整闭环。

### M2 接口联调

目标：用现有后端模块跑通真实数据。

范围：

- 学生登录。
- 作业列表。
- 提交作业。
- 提交详情。
- 结果查看。

验收：

- 学生提交能在教师端提交列表看到。
- 教师发布结果后学生端能看到。

### M3 可用性完善

目标：处理边界状态和移动端体验。

范围：

- 退回重交。
- 迟交。
- 上传异常。
- 空状态。
- 移动端布局。

验收：

- 常见异常均有明确提示和可恢复路径。

## 14. 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| 学生登录方式未定 | 影响真实上线 | MVP 先保留 dev-login 和作业链接回跳 |
| 文件上传未真实接入 | 影响附件作业 | 先支持文本和链接，附件走 upload-token |
| 学生误以为 AI 批阅可见 | 引发理解偏差 | 学生端文案只说教师批阅，不出现 AI 初评 |
| 教师修改作业后学生端不同步 | 提交错误 | 作业详情每次进入实时拉取 |
| 学生多次提交规则不清 | 影响状态设计 | MVP 建议仅退回后可重新提交 |

## 15. 推荐下一步

建议先做学生端静态原型和统一 mockData 闭环，顺序如下：

1. 在现有原型中新增学生端 6 个页面。
2. 扩展 `mockData`，加入学生和提交记录。
3. 让学生提交动作真实修改测试数据。
4. 让教师端提交统计读取同一份数据。
5. 让教师发布结果后学生结果页可见。

完成后，当前产品就具备教师端和学生端双角色的可演示闭环。
