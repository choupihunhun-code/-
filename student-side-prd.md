# 学生端教务系统 PRD

## 1. 文档定位

本文档定义教务系统学生端 MVP 的产品需求。学生端用于承接教师端已经规划的核心闭环：教师发布作业后，学生能够查看作业、完成提交、查看提交状态，并在教师发布后查看最终成绩与评语。

学生端第一阶段不做完整学习平台，不做课程社区、学习分析、题库考试或移动 App，而是优先补齐教师端闭环中缺失的学生操作侧。

## 2. 产品目标

### 2.1 核心目标

让学生用最低理解成本完成以下任务：

- 知道自己有哪些待完成作业。
- 看清作业要求、截止时间和提交方式。
- 上传附件、填写文本或链接并完成提交。
- 知道提交是否成功、是否迟交、是否被退回。
- 在教师发布结果后查看最终分数和教师评语。

### 2.2 业务目标

- 补齐教师端 MVP 的端到端闭环。
- 为教师端提交统计、AI 初评、教师复核和成绩导出提供真实数据来源。
- 降低学生漏交、错交、重复问询的概率。
- 保证 AI 初评只作为教师侧参考，不直接暴露给学生。

## 3. 产品范围定义

本阶段学生端的范围定义遵循“先做闭环，再做增强”的原则。第一版不追求大而全，而是先把教师端已经形成的作业管理链路补完整：教师发布作业后，学生能看到、能提交，教师批阅发布后，学生能查看结果。

### 3.1 先做什么

学生端第一阶段优先做能支撑主业务闭环的功能：

- 学生能够登录或通过作业链接进入系统。
- 学生能够查看自己需要完成的作业。
- 学生能够看清作业要求、截止时间和提交方式。
- 学生能够提交文本、链接或附件。
- 学生能够确认提交是否成功。
- 学生能够看到教师退回原因并重新提交。
- 学生能够在教师发布后查看最终成绩和评语。

这些能力直接决定教师端的提交统计、AI 初评、教师复核、结果发布和成绩导出是否能够成立，因此属于 MVP 必须完成的范围。

### 3.2 暂不做什么

学生端第一阶段不做完整学习平台，不做泛课程空间，也不做会显著拉长研发周期的复杂功能：

- 不做课程资料库。
- 不做课堂签到。
- 不做在线考试和题库。
- 不做小组作业协作。
- 不做学生互评。
- 不做作业讨论区。
- 不做复杂消息中心。
- 不做学习数据分析。
- 不做成绩申诉流程。
- 不做移动端原生 App。

这些功能可以提升体验或扩展业务场景，但不影响第一阶段“作业提交与结果查看”的核心闭环，因此暂不进入 MVP。

### 3.3 MVP 定义

学生端 MVP 是“最小可行提交与结果查看系统”。

MVP 成立的标准不是页面数量多，而是以下链路可以完整跑通：

```text
教师发布作业 -> 学生查看作业 -> 学生提交作业 -> 教师看到提交 -> 教师复核并发布 -> 学生查看结果
```

只要这条链路中任何一环缺失，学生端就不能算完成 MVP。

### 3.4 核心功能、辅助功能、后续功能

#### 核心功能

核心功能是没有它就无法完成主闭环的能力：

- 学生登录或身份校验。
- 作业列表。
- 作业详情。
- 作业提交。
- 提交成功与提交详情。
- 退回后重新提交。
- 教师发布后的结果查看。
- 学生本人数据权限控制。

#### 辅助功能

辅助功能用于提升提交效率、降低误操作和减少教师沟通成本：

- 作业状态筛选。
- 截止时间提醒。
- 迟交提示。
- 附件上传进度。
- 提交前确认。
- 空状态和异常提示。
- 作业链接登录后回跳。
- 移动端浏览器适配。

辅助功能不应喧宾夺主，优先服务“学生快速完成提交”。

#### 后续功能

后续功能是有价值但不影响 MVP 验证的能力：

- 学生消息通知中心。
- 成绩更正记录。
- 多版本提交对比。
- 学生申诉或留言。
- 作业历史归档高级筛选。
- 课程资料中心。
- 小组作业。
- 学生互评。
- 学习表现分析。

### 3.5 范围控制原则

- 第一版所有页面都围绕“作业”展开，不扩展成完整学生门户。
- 所有学生端状态都必须能对应到教师端的作业、提交、批阅和发布状态。
- 学生端不出现 AI 初评原始结果，避免学生误解 AI 是最终批阅者。
- 每新增一个功能，都要判断它是否帮助完成提交闭环；如果不能，默认后置。
- 先用静态测试数据跑通双端闭环，再进入接口联调。

## 4. 用户角色

### 4.1 学生

学生是学生端唯一核心用户。

学生可以：

- 登录或通过作业访问入口进入系统。
- 查看自己所在课程班的作业。
- 查看作业详情。
- 提交作业。
- 查看提交记录和退回原因。
- 在教师发布后查看最终成绩和评语。

学生不可以：

- 查看教师端 AI 初评原始建议。
- 查看其他学生提交。
- 修改课程班、作业、评分标准。
- 在教师未发布前查看成绩或评语。

### 4.2 教师

教师不是学生端直接用户，但教师端操作会影响学生端可见内容：

- 发布作业后，学生端显示作业。
- 修改截止时间或作业说明后，学生端同步显示。
- 退回修改后，学生端显示退回原因并允许重新提交。
- 发布成绩后，学生端显示最终分数和教师评语。

## 5. MVP 范围

### 5.1 必须包含

#### 登录与访问

- 支持学生账号登录。
- 支持开发或演示阶段的学生快速登录。
- 支持从课程班或作业链接进入。
- 登录后只能访问本人相关课程、作业、提交和结果。

#### 作业列表

- 展示学生本人相关课程班下的作业。
- 展示作业标题、课程班、截止时间、提交状态、批阅状态。
- 支持按状态查看：待提交、已提交、已退回、已发布结果。
- 支持进入作业详情、提交页、结果页。

#### 作业详情

- 展示作业标题、课程班、教师、截止时间、作业说明、附件、提交方式。
- 展示是否允许迟交。
- 展示当前提交状态。
- 学生不可见教师内部 AI 评分标准和 AI 原始建议。

#### 作业提交

- 支持文本提交。
- 支持链接提交。
- 支持附件提交。
- 至少需要提交文本、链接或附件中的一种。
- 提交前展示确认信息。
- 提交成功后展示提交时间、提交版本和状态。
- 截止后提交时标记为迟交。

#### 重新提交

- 教师退回后，学生可以查看退回原因。
- 学生可以基于退回原因重新提交。
- 重新提交后提交版本递增。
- 重新提交应重新进入教师端批阅或 AI 初评流程。

#### 提交状态查看

- 学生可以查看本人提交内容、附件、链接、提交时间。
- 学生可以看到状态：未提交、已提交、迟交、已退回、已重新提交。
- 学生可以看到批阅状态：待批阅、批阅中、已发布。

#### 结果查看

- 教师发布后，学生可以查看最终分数、教师评语、发布时间。
- 教师未发布前，学生看到“教师尚未发布结果”。
- 学生不可见 AI 建议分数、AI 原始评语、AI 评分依据。

### 5.2 暂不包含

- 课程资料库。
- 课堂签到。
- 在线考试和题库。
- 小组作业协作。
- 学生互评。
- 作业讨论区。
- 站内复杂消息中心。
- 学情分析报告。
- 学生申诉流程。
- 成绩趋势分析。
- 移动端原生 App。

## 6. 核心用户路径

### 6.1 提交作业路径

```text
学生登录 -> 作业列表 -> 作业详情 -> 填写/上传提交内容 -> 确认提交 -> 提交成功 -> 查看提交状态
```

### 6.2 查看结果路径

```text
学生登录 -> 作业列表 -> 已发布结果作业 -> 结果详情 -> 查看最终分数和教师评语
```

### 6.3 退回修改路径

```text
学生登录 -> 作业列表看到已退回 -> 查看退回原因 -> 修改提交内容 -> 重新提交 -> 等待教师复核
```

## 7. 业务流程梳理

本章节从学生进入产品到完成目标的角度，梳理学生端 MVP 的完整业务流程、参与角色、异常情况和系统支撑能力。

### 7.1 主流程：完成一次作业提交

目标：学生完成教师发布的作业提交，并让教师端能够继续进入 AI 初评和教师复核。

```text
教师发布作业
-> 系统向对应课程班学生开放作业
-> 学生登录或通过作业链接进入
-> 学生查看作业列表
-> 学生进入作业详情
-> 学生阅读作业说明、截止时间、提交方式
-> 学生填写文本、链接或上传附件
-> 系统校验提交内容
-> 学生确认提交
-> 系统生成或更新提交记录
-> 学生看到提交成功
-> 教师端提交统计同步变化
-> 作业进入 AI 初评或人工批阅流程
```

关键成功标准：

- 学生知道自己提交成功。
- 教师端能看到该学生从未提交变为已提交或迟交。
- 提交内容可被后续 AI 初评或教师批阅读取。

### 7.2 主流程：查看教师发布结果

目标：学生在教师完成复核并发布后查看最终成绩和评语。

```text
教师完成复核
-> 教师发布最终结果
-> 系统将结果状态改为学生可见
-> 学生登录后在作业列表看到已发布结果
-> 学生进入结果页
-> 学生查看最终分数、教师评语、发布时间
```

关键成功标准：

- 教师未发布前，学生看不到成绩和评语。
- 教师发布后，学生看到的是教师确认后的最终结果。
- 学生端不展示 AI 建议分数、AI 原始评语和 AI 评分依据。

### 7.3 分支流程：教师退回后重新提交

目标：教师发现提交不合格时退回，学生根据退回原因修改后重新提交。

```text
教师退回提交并填写原因
-> 学生作业列表显示已退回
-> 学生进入提交详情查看退回原因
-> 学生修改文本、链接或附件
-> 学生重新提交
-> 系统提交版本递增
-> 教师端重新进入待处理
```

关键成功标准：

- 学生能清楚看到为什么被退回。
- 重新提交不会覆盖历史版本到无法追踪。
- 教师端能识别这是重新提交，而不是首次提交。

### 7.4 参与角色

| 角色 | 参与环节 | 主要职责 |
| --- | --- | --- |
| 学生 | 登录、查看作业、提交作业、查看结果 | 完成本人作业提交，查看本人结果 |
| 教师 | 发布作业、查看提交、退回、复核、发布结果 | 定义作业要求，确认最终成绩和评语 |
| 系统 | 权限校验、状态流转、文件处理、数据同步 | 保证学生只能访问本人数据，并同步教师端统计 |
| AI | 提交后的初评建议 | 仅向教师提供建议，不直接向学生展示 |

### 7.5 异常情况

#### 登录与访问异常

- 学生账号不存在：提示联系教师确认名单。
- 登录验证码错误或过期：提示重新获取。
- 作业链接失效：提示从作业列表进入或联系教师。
- 学生不属于该课程班：禁止访问该作业。

#### 作业状态异常

- 作业已截止且不允许迟交：禁止提交，只允许查看详情。
- 作业已归档：默认不出现在待办列表，可在历史中查看。
- 作业被教师撤回或删除：学生端提示作业暂不可访问。

#### 提交异常

- 文本、链接、附件均为空：阻止提交。
- 附件上传失败：保留已填写内容，允许重试。
- 附件大小或格式不符合要求：提示限制规则。
- 网络中断或提交失败：不清空表单，提示重试。
- 重复点击提交：按钮进入处理中状态，防止重复提交。

#### 结果查看异常

- 教师尚未发布结果：展示待批阅，不返回分数和评语。
- 结果已更正：展示最新结果，并可标记“已更正”。
- 学生尝试访问他人结果：禁止访问。

### 7.6 支撑能力

为保证业务流程成立，系统需要以下支撑能力：

#### 身份与权限

- 学生登录或作业链接身份校验。
- 学生、课程班、作业之间的归属校验。
- 学生仅能访问本人提交和结果。

#### 作业可见性

- 只向学生展示已发布作业。
- 草稿作业学生不可见。
- 作业截止、关闭、归档状态需要影响学生端操作。

#### 提交能力

- 文本、链接、附件三类提交方式。
- 提交内容校验。
- 附件上传和下载。
- 提交版本记录。
- 迟交判断。

#### 状态同步

- 学生提交后，教师端提交统计实时或准实时更新。
- 教师退回后，学生端显示退回状态和原因。
- 教师发布后，学生端结果可见。

#### 通知与反馈

- 提交成功反馈。
- 上传失败反馈。
- 迟交提示。
- 退回原因提示。
- 结果发布状态提示。

#### 数据安全

- 结果发布前不返回成绩字段。
- AI 初评结果不返回给学生端。
- 附件访问需要鉴权。
- 所有学生端接口按当前学生身份过滤数据。

## 8. 页面范围

### 8.1 学生登录页

页面目标：让学生快速进入自己的作业空间。

核心元素：

- 学号或手机号输入。
- 验证码或密码输入。
- 登录按钮。
- 作业链接访问提示。

关键规则：

- 登录成功后进入学生作业列表。
- 如果通过作业链接进入，登录后优先跳转到对应作业详情。
- 登录失败时提示账号不存在、验证码错误或登录过期。

### 8.2 学生首页 / 作业列表页

页面目标：让学生知道现在要做什么。

核心元素：

- 待提交数量。
- 已提交数量。
- 已退回数量。
- 已发布结果数量。
- 作业列表。
- 状态筛选。

作业卡片字段：

- 作业标题。
- 课程班名称。
- 教师姓名。
- 截止时间。
- 提交状态。
- 批阅状态。
- 操作按钮。

主要操作：

- 查看详情。
- 继续提交。
- 查看提交。
- 查看结果。

### 8.3 作业详情页

页面目标：让学生完整理解作业要求。

核心元素：

- 作业标题。
- 课程班。
- 截止时间。
- 作业说明。
- 作业附件。
- 提交方式。
- 是否允许迟交。
- 当前状态。

主要操作：

- 进入提交页。
- 查看已提交内容。
- 查看结果。

关键规则：

- 草稿作业不可见。
- 已归档作业默认不出现在待办，但可在历史中查看。
- 教师内部 AI 评分标准不向学生展示。

### 8.4 作业提交页

页面目标：让学生顺利完成提交。

核心元素：

- 文本输入框。
- 链接输入框。
- 附件上传区。
- 提交说明。
- 截止时间提示。
- 提交按钮。

校验规则：

- 文本、链接、附件至少填写一种。
- 链接格式必须合法。
- 附件大小和格式需符合系统限制。
- 作业关闭且不允许迟交时，禁止提交。
- 迟交时提交按钮旁展示迟交提示。

异常状态：

- 上传失败。
- 文件过大。
- 文件类型不支持。
- 网络中断。
- 作业已关闭提交。

### 8.5 提交成功页

页面目标：让学生明确知道提交已完成。

核心元素：

- 提交成功状态。
- 提交时间。
- 提交版本。
- 是否迟交。
- 已提交内容摘要。
- 返回作业列表按钮。
- 查看提交详情按钮。

### 8.6 提交详情页

页面目标：让学生回看自己的提交记录。

核心元素：

- 提交文本。
- 提交链接。
- 附件列表。
- 提交时间。
- 提交状态。
- 退回原因。
- 版本记录。

主要操作：

- 下载或查看附件。
- 教师退回后重新提交。

### 8.7 批阅结果页

页面目标：让学生查看教师发布后的最终结果。

核心元素：

- 最终分数。
- 教师评语。
- 发布时间。
- 作业信息。
- 提交信息。

关键规则：

- 只展示教师发布后的结果。
- 不展示 AI 建议分数。
- 不展示 AI 原始评语。
- 不展示教师草稿。

## 9. 状态定义

### 9.1 学生视角作业状态

| 状态 | 说明 | 学生可操作 |
| --- | --- | --- |
| 待提交 | 作业已发布，学生未提交 | 查看详情、提交 |
| 已提交 | 学生已提交，教师未发布结果 | 查看提交 |
| 迟交 | 学生在截止后提交 | 查看提交 |
| 已退回 | 教师退回修改 | 查看原因、重新提交 |
| 已重新提交 | 学生退回后再次提交 | 查看提交 |
| 已发布结果 | 教师已发布成绩和评语 | 查看结果 |

### 9.2 结果可见性

## 11. 接口需求

学生端 MVP 复用或补充以下接口：

- `POST /api/auth/student/dev-login`
- `GET /api/student/assignments`
- `GET /api/student/assignments/{assignment_id}`
- `POST /api/student/assignments/{assignment_id}/submit`
- `GET /api/student/submissions/{submission_id}`
- `GET /api/student/assignments/{assignment_id}/result`
- `POST /api/files/upload-token`

关键接口规则：

- 所有学生接口只返回当前学生本人数据。
- 作业列表只返回已发布且学生所属课程班可见的作业。
- 提交接口至少要求文本、链接或附件之一。
- 结果接口只返回教师已发布结果。
- 结果接口不得返回 AI 原始建议。

## 12. 权限与安全

- 学生只能访问本人作业、提交和结果。
- 学生不能通过修改 ID 访问其他学生提交。
- 学生不能访问教师端接口。
- 学生不能查看 AI 初评详情。
- 附件下载需校验学生身份和作业归属。
- 结果未发布时，接口返回未发布状态，不返回分数和评语。

## 13. 验收标准

学生端 MVP 完成时，应满足：

- 学生可以登录并进入作业列表。
- 学生可以看到教师已发布给自己的作业。
- 学生可以查看作业详情。
- 学生可以通过文本、链接或附件完成提交。
- 提交后教师端提交统计会发生变化。
- 截止后提交会被标记为迟交。
- 教师退回后，学生可以看到退回原因并重新提交。
- 教师发布前，学生看不到成绩和 AI 初评。
- 教师发布后，学生可以看到最终分数和教师评语。
- 学生无法访问其他学生数据。

## 14. 成功指标

- 学生首次进入后 1 分钟内能找到待提交作业。
- 学生提交成功率达到 95% 以上。
- 因“不知道是否提交成功”产生的咨询显著减少。
- 教师端未提交提醒后，学生补交转化率可被统计。
- 学生端结果查看不引发 AI 建议外泄。

## 15. 待确认问题

- 学生正式登录方式采用学号密码、手机号验证码，还是作业链接加身份校验。
- 是否允许学生在教师未退回时主动覆盖提交。
- 单个作业允许上传的附件格式和大小上限。
- 截止后是否默认允许迟交。
- 教师发布成绩后是否允许学生看到更正记录。
- 是否需要学生端消息通知，还是第一版只做页面状态。
